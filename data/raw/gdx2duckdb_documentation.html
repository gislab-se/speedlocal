1. Overview

This system provides tools for importing TIMES (The Integrated MARKAL-EFOM System) energy model outputs into a DuckDB database for analysis and visualization. It consists of three main components:

  1. Database Creation Script (R) - Imports GDX files into DuckDB
  2. Visualization Application (Python/Streamlit) - Interactive web-based data explorer
  3. Legacy Shiny Application (R) - Previous CSV-based visualization tool
Purpose: The system converts complex TIMES model outputs from GDX format into an accessible, queryable database format with accompanying visualization tools for energy system analysis.

  1. Input: Multiple TIMES model GDX files
  2. Processing: R script extracts data and metadata
  3. Storage: Data organized in DuckDB relational database
  4. Analysis: Query via SQL or visualization tools

3. Database Creation Script (R)

File: create_times_db_multiple_scenarios.R

3.1 Purpose

Converts multiple TIMES model GDX output files into a structured DuckDB database with proper relationships and metadata.

3.2 Key Features

  • Processes multiple GDX files in batch
  • Extracts descriptive metadata (sectors, commodities, processes, etc.)
  • Creates relational database structure with foreign keys
  • Tracks data provenance (source files, timestamps)
  • Supports scenario-model relationships via modelname set
  • Includes automated testing functionality

Description Tables

These tables provide human-readable descriptions for coded identifiers:

  • sector_desc - Sector descriptions
  • subsector_desc - Subsector descriptions
  • service_desc - Service descriptions
  • techgroup_desc - Technology group descriptions
  • comgroup_desc - Commodity group descriptions
  • topic_desc - Topic descriptions
  • prc_desc - Process descriptions
  • com_desc - Commodity descriptions
  • attr_desc - Attribute descriptions

Each description table has: id (VARCHAR, PK), description (VARCHAR), scen (VARCHAR, PK)

3.5 Key Functions

read_gdx_set(container, set_name, scen)

Extracts a set from a GDX container and converts it to a dataframe.

Parameters:

  • container - GAMS container object
  • set_name - Name of the set to extract
  • scen - Scenario identifier

Returns: Dataframe with id, description, and scen columns

Special Handling:

  • prc_desc and com_desc - Removes first column (region) and deduplicates
  • Single-column sets - Creates id-only dataframe
  • Multi-column sets - Creates id and description columns

read_modelname_set(container)

Extracts scenario-to-TIMES-model relationships.

Returns: Dataframe with scen and timesmodel columns

load_gdx_files(gdx_files, db_path)

Main function that processes all GDX files and creates the database.

Process:

  1. Remove existing database
  2. Create new database with schema
  3. For each GDX file:
    • Read container
    • Extract scenario name
    • Log file metadata
    • Process modelname set (if present)
    • Process description sets
    • Import timesreport data
  4. Return connection

3.6 Usage Example

R Usage
# Load required libraries
library(gamstransfer)
library(dplyr)
library(DBI)
library(duckdb)

# Configure paths
gdx_files <- c(
    "./GDX/scenario1_TIMESreport.gdx",
    "./GDX/scenario2_TIMESreport.gdx"
)

db_path <- "./duckDB/times_db.duckdb"

# Create database
con <- load_gdx_files(gdx_files, db_path)

# Query example
results <- dbGetQuery(con, "
    SELECT 
        f.scen,
        f.sector,
        s.description as sector_desc,
        f.year,
        SUM(f.value) as total_value
    FROM timesreport f
    LEFT JOIN sector_desc s ON f.sector = s.id AND f.scen = s.scen
    WHERE f.topic = 'energy'
    GROUP BY f.scen, f.sector, s.description, f.year
")

# Close connection
dbDisconnect(con, shutdown = TRUE)

4.Query Examples

4.1 Basic Data Retrieval

SQL Query
SELECT * 
FROM timesreport 
WHERE scen = 'BASE' 
  AND topic = 'energy' 
  AND year = '2030'
LIMIT 10;

4.2 Energy by Sector with Descriptions

SQL Query
SELECT 
    f.scen,
    f.sector,
    s.description as sector_name,
    f.year,
    SUM(f.value) as total_energy,
    f.unit
FROM timesreport f
LEFT JOIN sector_desc s 
    ON f.sector = s.id AND f.scen = s.scen
WHERE f.topic = 'energy'
  AND f.attr = 'f_in'
GROUP BY f.scen, f.sector, s.description, f.year, f.unit
ORDER BY f.year, f.sector;

4.3 Scenario Comparison

SQL Query
SELECT 
    f.scen,
    f.year,
    f.comgroup,
    c.description as commodity,
    SUM(f.value) as value,
    f.unit
FROM timesreport f
LEFT JOIN comgroup_desc c 
    ON f.comgroup = c.id AND f.scen = c.scen
WHERE f.topic = 'energy'
  AND f.sector = 'ELC'
  AND f.scen IN ('BASE', 'SCENARIO_A')
GROUP BY f.scen, f.year, f.comgroup, c.description, f.unit
ORDER BY f.year, f.scen;

4.4 Model-Scenario Relationships

SQL Query
SELECT 
    sm.scen,
    sm.timesmodel,
    sf.filename,
    sf.load_timestamp
FROM scenario_model sm
JOIN source_files sf ON sm.file_id = sf.file_id
ORDER BY sm.timesmodel, sm.scen;

6.Troubleshooting

6.1 Common Issues

"Could not remove existing database file"

  • Close all connections to the database
  • Check if other processes are using the file
  • Ensure you have write permissions

"No data matches the selected filters"

  • Verify scenario names match exactly
  • Check that topics and attributes exist in data
  • Review excluded sectors list

Empty visualizations

  • Confirm database connection is successful
  • Verify GDX files contain expected data structure
  • Check that timesreport table has data

6.2 File Paths

  • Use forward slashes (/) or double backslashes (\\) in Windows
  • Relative paths are relative to script location
  • Ensure GDX files are accessible

7. Best Practices

7.1 Database Management

  • Keep backups of original GDX files
  • Document scenario names and purposes
  • Use descriptive database names with dates

7.2 Performance

  • Index frequently queried columns
  • Use specific queries instead of SELECT *
  • Close connections when finished

7.3 Data Quality

  • Validate GDX files before import
  • Review test output after database creation
  • Check for missing descriptions

7.4 Version Control

  • Track configuration changes
  • Document GDX file versions
  • Maintain changelog for database updates
